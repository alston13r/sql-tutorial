Module 15

Moving Data Between Tables

All exercises in this lesson have been completed.

Copy rows from one table into another with INSERT INTO ... SELECT — useful for archives, backups, and bulk transfers without retyping values.

Overview

Module 4 taught single-row INSERT statements where you list each value by hand. That works for one new employee, but it is a poor fit when you already have matching rows in another table — for example, finished tasks you want to keep in an archive.

INSERT INTO ... SELECT reads rows from a source query and writes them into a target table in one step. The column lists must line up: each selected expression fills the matching target column.

After a successful archive copy, some apps also DELETE the originals from the active table. You already know DELETE from Module 4; this module focuses on the copy step.

INSERT INTO SELECT

The pattern is: name the target table and columns, then supply a SELECT that returns one value per column for each row to insert.

Example — archive every Planned task and stamp the archive date in the SELECT list: 

INSERT INTO tasks_archive (id, project_id, title, archived_on)
SELECT id, project_id, title, '2024-11-01'
FROM tasks
WHERE status = 'Planned';

Notice archived_on is not a column on tasks. The SELECT can mix source columns with literal values. SQLite does not use SELECT INTO the way some other databases do; use INSERT INTO ... SELECT instead.

Filtered copies

The WHERE clause on the SELECT controls which source rows move. You can filter on status, project, date ranges, or any other condition you would use in a normal query.

Example — copy only in-progress work for project 3: 

INSERT INTO tasks_archive (id, project_id, title, archived_on)
SELECT id, project_id, title, '2024-11-15'
FROM tasks
WHERE status = 'In Progress' AND project_id = 3;

Always list target columns explicitly when the tables do not share the same shape. That keeps the mapping clear and avoids inserting into the wrong column.

CREATE TABLE AS

Sometimes you want a brand-new table filled from a query, instead of inserting into a table that already exists. SQLite supports that with CREATE TABLE ... AS SELECT.

This is useful for snapshots. For example, using the familiar employees table, you can build a department summary that stores the average salary of current employees in each department: 

CREATE TABLE department_avg_salaries AS
SELECT department, AVG(salary) AS avg_salary
FROM employees
GROUP BY department;

That creates department_avg_salaries and fills it in one step. Column names come from the SELECT list. Unlike a carefully designed CREATE TABLE from Module 11, this form does not copy constraints such as primary keys — it is best for snapshots and working copies.

Summary

  • INSERT INTO ... SELECT copies query results into another table without listing each value by hand.
  • Target columns and selected expressions must match in order and count.
  • Use WHERE on the SELECT to archive or copy only the rows you want.
  • Literals in the SELECT list can fill columns the source table does not have, such as archived_on.
  • CREATE TABLE ... AS SELECT creates a new table and fills it from a query in one step — useful for department summaries and other snapshots.
