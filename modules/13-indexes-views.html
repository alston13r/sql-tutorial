Module 13

Indexes & Views

All exercises in this lesson have been completed.

Indexes help SQLite find rows faster. Views save a query under a name so you can reuse it like a table.

Overview

Earlier modules focused on storing and querying data. This module adds two tools that sit on top of tables you already have:

  • An index is a lookup structure on one or more columns. It can make WHERE and ORDER BY work faster on large tables.
  • A view is a saved SELECT. It does not store its own copy of the rows; it runs the query whenever you use the view.

Primary keys already give SQLite a fast way to find rows by id. Extra indexes are useful when you often filter or sort on other columns.

CREATE INDEX

CREATE INDEX builds an index on a table column. You choose a name for the index and the column (or columns) it covers.

Example — speed up lookups by department: 

CREATE INDEX idx_employees_department
ON employees(department);

Indexes help reads, but they also take space and can make inserts and updates a little slower because SQLite must keep the index up to date. Use them on columns you filter or sort by often. To remove an index later, use DROP INDEX index_name;.

Views

A view is a named query. After you create it, you can SELECT from the view the same way you select from a table.

Example — a reusable list of high earners: 

CREATE VIEW high_earners AS
SELECT name, department, salary
FROM employees
WHERE salary >= 90000;

Then you can run: 

SELECT * FROM high_earners;

The view does not copy those rows into a new table. Each time you query it, SQLite runs the saved SELECT against the current employees data.

Summary

  • CREATE INDEX builds a lookup structure on columns you filter or sort by often.
  • Indexes can speed up reads, but they use space and add a little cost to writes.
  • CREATE VIEW saves a SELECT under a name you can query like a table.
  • Views do not store a separate copy of the data; they re-run the query each time.
