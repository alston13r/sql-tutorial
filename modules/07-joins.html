Joins

All exercises in this lesson have been completed.

So far, each employee row stored the department name as text. This module uses a redesigned schema where employees reference departments by department_id — and joins combine the related rows when you query.

Relating Tables

In earlier modules, the employees table had a department column with values like 'Engineering'. That is simple to query, but the department name is repeated on every employee row, and extra department details (like budget) live in a separate departments table.

A more relational design stores a reference instead: 

employees                          departments
─────────                          ───────────
id                                 id
name                               name
department_id  ──matches id──►      budget
job_title
salary
manager_id

Each employee's department_id points at a row in departments. The employee row no longer contains the department name directly — so a query like SELECT name, department FROM employees no longer works. You need a join to line up matching rows from both tables.

Joins use an ON clause to define how rows match. Table aliases (such as e and d) keep column names short when the same name appears in both tables: 

SELECT e.name, d.name AS department
FROM employees e
JOIN departments d ON e.department_id = d.id;

This practice database has the same 14 employees as before, plus a Research department with no employees yet. A manager_id column on employees links some employees to their manager (covered in the self-join section).

INNER JOIN

An INNER JOIN returns only rows that match on both sides. If an employee has no matching department, or a department has no employees, those rows are left out.

To list every employee with their department name: 

SELECT e.name AS employee, d.name AS department
FROM employees e
INNER JOIN departments d ON e.department_id = d.id;

Result: 14 rows — every employee has a valid department_id. The Research department does not appear because it has no employees.

LEFT JOIN

A LEFT JOIN (short for LEFT OUTER JOIN) keeps every row from the left table and attaches matching data from the right when it exists. If there is no match, the right-hand columns are NULL.

To list every department and any employee assigned to it: 

SELECT d.name AS department, e.name AS employee
FROM departments d
LEFT JOIN employees e ON e.department_id = d.id;

Result: 15 rows. Departments with several employees appear on multiple rows.

Self Join

A self join joins a table to itself. You use two aliases for the same table so SQL can treat them as separate sides of the join.

Here, manager_id on employees references another employee's id. To list each employee alongside their manager's name: 

SELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

A LEFT JOIN keeps employees with no manager — their manager column is NULL. Bob Martinez reports to Alice Chen; Eve Johnson reports to David Kim.

Summary

Joins combine related rows from two tables (or the same table twice, in a self join). The join type decides which rows make it into the result:

  • INNER JOIN returns only the intersection — rows that match on both sides. Unmatched rows from either table are dropped. In this module, that is why Research does not appear in an inner join between departments and employees.
  • LEFT JOIN keeps every row from the left table and appends matching columns from the right when a match exists. When there is no match, the right-hand columns are NULL.

Which table you put on the left matters for a LEFT JOIN. Starting from departments lists every department; starting from employees would list every employee instead.
