Constraints & Keys

All exercises in this lesson have been completed.

Module 11 defined table structure. This module adds rules that protect that structure by limiting what data can be inserted.

Constraints Overview

Constraints and keys enforce data integrity. When you try to insert invalid data, SQLite stops the statement and returns an error instead of writing the bad row.

Common failures include inserting NULL into a NOT NULL column, duplicating a UNIQUE or primary key value, referencing a missing parent row with a foreign key, or failing a CHECK condition.

NOT NULL and UNIQUE

NOT NULL means a value is required. UNIQUE means a value may appear only once in that column. If you try to insert a missing or duplicate value, SQLite raises an error and the row is not inserted.

Example: 

CREATE TABLE users (
  id INTEGER,
  username TEXT NOT NULL UNIQUE,
  email TEXT UNIQUE
);

Primary Key

A primary key uniquely identifies each row. A single-column primary key often uses one id column, but SQLite also supports composite primary keys across multiple columns.

In practice, a primary key is usually the table's main identity column or identity column set. Other tables can reference it with foreign keys, and it gives the database one clear way to tell rows apart.

You could sometimes combine UNIQUE and NOT NULL to enforce similar rules on a column, but a primary key has a more specific job: it marks the official identifier for the row. Each table has one primary key definition, even if that definition uses multiple columns.

Single-column example: 

CREATE TABLE departments (
  id INTEGER PRIMARY KEY,
  name TEXT
);

Composite example: 

CREATE TABLE enrollments (
  student_id INTEGER,
  course_id INTEGER,
  PRIMARY KEY (student_id, course_id)
);

If you try to insert a duplicate primary key value, SQLite rejects the insert.

Foreign Key

A foreign key links a column in one table to a key in another table. Like primary keys, foreign keys can be single-column or composite.

Single-column example: 

CREATE TABLE orders (
  id INTEGER PRIMARY KEY,
  customer_id INTEGER,
  FOREIGN KEY (customer_id) REFERENCES customers(id)
);

Composite example: 

CREATE TABLE course_sessions (
  department TEXT,
  course_number INTEGER,
  term TEXT,
  FOREIGN KEY (department, course_number)
    REFERENCES courses(department, course_number)
);

If you insert a foreign key value that does not match a parent row, meaning a parent row with that key does not exist, SQLite rejects the insert.

CHECK and DEFAULT

CHECK enforces a condition, and DEFAULT supplies a value when you do not provide one.

Example: 

CREATE TABLE accounts (
  id INTEGER PRIMARY KEY,
  status TEXT DEFAULT 'active',
  balance REAL CHECK (balance >= 0)
);

A row that fails the CHECK condition is rejected. A missing status value uses the default.

Tip: In SQLite, you can inspect a table's schema before inserting rows. These two commands are especially useful when you want to confirm a DEFAULT value or other constraints on users: 
SELECT sql
FROM sqlite_master
WHERE type = 'table' AND name = 'users';
PRAGMA table_info('users');

AUTOINCREMENT

SQLite can auto-generate integer ids when a column is declared as INTEGER PRIMARY KEY AUTOINCREMENT. This is most useful when you do not want to supply ids manually. 

CREATE TABLE logs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  message TEXT
);

Each new row gets the next integer id automatically. You usually do not include the id in your INSERT statement. If you do provide an id yourself, SQLite uses that value for the row, and the next auto-generated id will continue from the largest id that has been used so far.

Summary

  • NOT NULL requires a value.
  • UNIQUE prevents duplicates.
  • PRIMARY KEY uniquely identifies rows, with both single-column and composite forms.
  • FOREIGN KEY links rows to valid parent rows, including composite keys.
  • CHECK rejects values that fail a rule.
  • DEFAULT fills in a value when you do not provide one.
  • AUTOINCREMENT lets SQLite assign integer ids automatically.
