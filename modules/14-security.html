Module 14

Security Best Practices

All exercises in this lesson have been completed.

When an app builds SQL from user input, attackers can change the meaning of the query. This module shows how that happens and how parameterized queries prevent it.

Overview

So far you have written SQL yourself. Real applications often take values from a form — a username, a password, a search box — and put them into a query. If those values are pasted into the SQL string with concatenation, the user can type SQL syntax instead of ordinary text.

That attack is called SQL injection. The fix is to keep user values out of the SQL text and bind them as parameters instead.

SQL Injection

Imagine a login page that builds a query like this in application code. In JavaScript, that often looks like string concatenation: 

const sql =
  "SELECT username FROM users " +
  "WHERE username = '" + username + "' " +
  "AND password = '" + password + "';";

In Python, the same unsafe pattern often uses an f-string or other template string: 

sql = f"""
SELECT username FROM users
WHERE username = '{username}' AND password = '{password}';
"""

If someone types a normal username and password, that works. If they type SQL into the fields, they can change the WHERE clause — for example by closing the quote early and commenting out the password check.

Passwords in this demo are stored as plain text so the lesson stays simple. Real apps should store hashed passwords. Injection can also lead to reading, changing, or deleting data beyond a login form.

SQL Comments

SQL comments let you ignore part of a statement. Attackers often use them during injection to disable the rest of a query after they close a quote.

SQLite supports two common comment styles:

  • -- starts a line comment. Everything from -- to the end of that line is ignored.
  • /* ... */ is a block comment. Everything between the markers is ignored, even across multiple lines.

Example — a username of admin'-- can turn this: 

SELECT username FROM users
WHERE username = 'admin'--' AND password = 'nope';

into a query that only checks the username. The password condition is commented out, so the login no longer requires the real admin password.

Parameterized Queries

A parameterized (or prepared) query keeps the SQL structure fixed and sends values separately. The database treats those values as data, not as SQL syntax.

The same login, written safely in JavaScript with sql.js-style placeholders: 

db.prepare(
  "SELECT username FROM users WHERE username = ? AND password = ?"
).get(username, password);

If the username is admin'--, it is compared as a literal string. It does not close a quote or comment out the password check. The mock login above is intentionally vulnerable; real apps should use parameters (and hashed passwords).

Summary

  • SQL injection happens when user input is concatenated or interpolated into a query and changes the SQL structure.
  • The same risk shows up in JavaScript concatenation and Python template strings.
  • SQL comments (-- and /* */) are often used in injection payloads to ignore the rest of a clause, such as a password check.
  • Parameterized queries keep values separate from the SQL text so quote tricks are treated as ordinary data.
  • Real apps should also hash passwords; this module used plain text only for the demo.
